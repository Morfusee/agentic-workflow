1. IAM & AWS CLI

Core IAM Concepts
ConceptDescriptionKey Exam Point
UserIdentity for a person or serviceLong-term credentials (password + access keys)
GroupCollection of usersNot a true principal — can't be referenced in a policy's Principal element
RoleIdentity with temporary credentialsUse for EC2, cross-account access, federated users. Issued via STS
PolicyJSON document defining permissionsIdentity-based (attached to user/group/role) or Resource-based (attached to resource)
Policy Structure
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow" | "Deny",
    "Action":   ["service:operation", ...],
    "Resource": "arn:aws:...",
    "Condition": {              // optional
      "IpAddress": { "aws:SourceIp": "10.0.0.0/8" }
    }
  }]
}
IAM Evaluation Logic

Order: Default Deny → Allow → Explicit Deny (wins all)

An explicit Deny always overrides any Allow. This is the most tested exam concept in IAM.

EC2 Purchase Options
OptionUse WhenKey Exam Note

Dedicated HostCompliance / server-bound licensingPhysical server, visibility into sockets/cores. Most expensive

Placement Groups
TypeStrategyUse CaseLimitation
ClusterSame AZ, low latencyHPC, tightly-coupled appsSingle AZ — failure risk
SpreadDistinct hardware per instanceCritical instances (max 7 per AZ)Limited to 7 instances per AZ
PartitionGroups partitioned across racksLarge distributed apps (Hadoop, Cassandra)Partitions share no hardware within partition
Other Key Facts
  • Public IP changes on stop/start (unless Elastic IP)
  • ENI = virtual network card, can be detached and reattached (failover)
  • Hibernate = RAM state written to EBS root volume. Preserves in-memory state
  • User Data = bootstrap script run at launch
  • Instance store = ephemeral, high IOPS, data lost on stop/terminate

3. EC2 SAA-Level

Key SAA-Level Concepts
TopicDetails
Nitro SystemUnderlying hardware for modern EC2. Dedicated hardware for networking, storage, security. Enables bare-metal-like performance, EBS encryption on instances with no additional cost, faster networking (ENA), and NVMe local storage.
AMIAmazon Machine Image — template for an EC2 instance. Contains OS, software, block device mappings. Can be shared across accounts. Based on EBS snapshots of root volume.
EBS EncryptionUses KMS. Encrypted snapshots → encrypted volumes. Unencrypted snapshots must be copied with encryption to attach to encrypted instances.
Instance MetadataAccessible at http://169.254.169.254/latest/meta-data/. Contains instance info (not user data). No auth required — role credentials are available here.

4. ELB & Auto Scaling Groups

Load Balancer Types
TypeLayerKey FeaturesBest For
ALBLayer 7 (HTTP/HTTPS)Path-based routing, host-based routing, HTTP/2, gRPC, redirects, fixed response, OIDC auth, WAF integrationWeb apps, microservices, containers
NLBLayer 4 (TCP/UDP/TLS)Static IP per AZ, highest performance (millions req/s), preserves source IP, TLS terminationTCP/UDP traffic, extreme scale, static IP needs
GLBLayer 3/4 (IP)Transparent network gateway + LB. Route traffic to third-party appliances (firewalls, IDS/IPS)Security appliance integration (GENEVE protocol)
ALB Features in Detail
FeatureWhat It DoesExam Tip
Target GroupsEC2 instances, Lambda, IP, ECS tasksALB routes to target groups, not directly to instances
Sticky SessionsCookie-based routing to same targetDuration-based cookie or application cookie. Can cause uneven load
Cross-zone LBDistributes traffic evenly across all instances, not just within AZALB = on by default (free). NLB = off by default (charged)
Connection DrainingWait for in-flight requests to complete before deregistering targetAlso called "deregistration delay." Default 300s, can be 0–3600
SSL TerminationALB handles SSL decryption, instance gets plain HTTPUse ACM certificates. Must have listeners with TLS certs
Auto Scaling Groups (ASG)
ConceptDetails
Min / Desired / MaxASG always tries to keep running == Desired, within [Min, Max]
Launch Template (preferred)AMI, instance type, SG, key pair, user data. Newer than Launch Config (no updates on LC)
Health checksEC2 status + ELB health checks. Unhealthy instances are terminated and replaced
Scaling Policies
TypeHow It WorksWhen to Use
Target TrackingSet target value (e.g., CPU=50%), ASG auto-creates alarmsSimplest — most use cases
Step ScalingCloudWatch alarm + step adjustments (e.g., +2 if CPU > 70%, +4 if > 90%)Need different reactions for different breach sizes
Simple ScalingSingle adjustment per alarm breach. Must wait for cooldownLegacy. Step scaling is better
ScheduledScale at specific timesPredictable traffic patterns (end of month, sales)
Scaling for SAA Exam: Key Rules
  • ASGs cannot scale-in below Min or above Max
  • Cooldown period prevents rapid scaling before metrics stabilize (default 300s)
  • Lifecycle hooks let you run custom actions before instance launches or terminates
  • ASG with ALB: ASG + ALB + Target Group = the trifecta. ASG auto-registers instances with the target group
