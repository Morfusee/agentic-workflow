Lesson 2: NAT & Port Forwarding

Giving VMs internet access through your server's single public IP — without buying extras.

Why this replaces Additional IPs

OVH gave your KimSufi one public IP. If you want three VMs serving web apps, you have two options:

  1. Buy Additional IPs (€2–3/mo each) — each VM gets its own public IP. Simple, but costs add up.
  2. Use NAT + port forwarding (free) — VMs share the server's IP. Outbound traffic is masqueraded (all VMs can reach the internet). Inbound traffic is forwarded by port number (port 443 goes to VM1, port 51820 to VM2, etc.).

You chose option 2. This is exactly how your home router works: one public IP from your ISP, all your devices behind it.

The topology

        Internet
           |
   Public IP: 51.195.46.19
           |
   [Proxmox Host]
   [vmbr0 bridge: 192.168.0.1/24]
   [iptables: MASQUERADE all outbound]
   [iptables: DNAT selected ports inbound]
     /          |          \
   VM1         VM2         VM3
  192.168.0.2 192.168.0.3 192.168.0.4
  port 443     port 51820  port 80
  (Nextcloud)  (WireGuard) (static site)

Each VM uses a private IP on the bridge. The host runs iptables rules to:

  • MASQUERADE (SNAT): rewrite the source IP of outgoing VM packets → the internet sees them as coming from 51.195.46.19
  • DNAT (port forward): rewrite the destination of incoming packets on specific ports → they land on the right VM

How NAT works (conceptually)

Outbound traffic (VM → internet)

  1. VM1 sends a packet to 8.8.8.8:53 (DNS query). Source: 192.168.0.2:12345.
  2. The host receives it on vmbr0. The MASQUERADE rule rewrites source to 51.195.46.19:54321. The kernel stores this mapping in conntrack.
  3. The internet sees the query coming from your server IP. The response goes back to 51.195.46.19:54321.
  4. The host looks up conntrack, reverses the mapping, and delivers the response to 192.168.0.2:12345. VM1 gets its answer.

Inbound traffic (internet → VM service)

  1. A user connects to 51.195.46.19:443 (HTTPS).
  2. The DNAT rule matches: "packet arriving on port 443 → send to 192.168.0.2:443".
  3. The kernel rewrites the destination and delivers it to VM1.
  4. VM1's web server responds. The MASQUERADE rule rewrites the source, and conntrack ensures the reply reaches the original user.

The actual config

On your Proxmox host, you need three things:

1. Network interfaces (/etc/network/interfaces)

Assuming a single-NIC KimSufi server (no bonding):

auto lo
iface lo inet loopback

auto ensXXX             # replace with your actual NIC name
iface ensXXX inet static
        address 51.195.46.19/32    # your server's public IP
        gateway 100.64.0.1         # OVH's gateway
        post-up echo 1 > /proc/sys/net/ipv4/ip_forward
        post-up echo 1 > /proc/sys/net/ipv4/conf/vmbr0/proxy_arp

auto vmbr0
iface vmbr0 inet static
        address 192.168.0.1/24     # bridge IP
        bridge-ports ensXXX        # attach the physical NIC
        bridge-stp off
        bridge-fd 0
OVH-specific quirks
  • The public IP is /32 — it doesn't share a subnet with anything. The gateway (100.64.0.1) is on a carrier-grade NAT network.
  • proxy_arp is often needed on OVH for the host to respond to ARP queries for the bridge subnet.
  • If you can't reach your VMs after setup, check whether you need net.ipv4.conf.vmbr0.proxy_arp = 1.
  • To find your NIC name: run ip link on the server. It'll be something like enp2s0 or eno1.

2. IP forward (already in the post-up above)

Also add to /etc/sysctl.conf to survive reboots:

net.ipv4.ip_forward = 1
sysctl -p

3. iptables rules — the NAT section

These handle both outbound (MASQUERADE) and inbound (port forwarding):

# Outbound: all traffic from VMs gets the server's public IP
iptables -t nat -A POSTROUTING -s 192.168.0.0/24 -o ensXXX -j MASQUERADE

# Inbound: forward specific ports to specific VMs
iptables -t nat -A PREROUTING -i ensXXX -p tcp --dport 443 -j DNAT --to-destination 192.168.0.2:443
iptables -t nat -A PREROUTING -i ensXXX -p udp --dport 51820 -j DNAT --to-destination 192.168.0.3:51820

# Allow forwarding between interfaces
iptables -A FORWARD -i vmbr0 -o ensXXX -j ACCEPT
iptables -A FORWARD -i ensXXX -o vmbr0 -m state --state RELATED,ESTABLISHED -j ACCEPT

4. Making iptables persistent

These rules disappear on reboot. Save them:

apt install iptables-persistent
netfilter-persistent save

Or manually:

iptables-save > /etc/iptables/rules.v4

5. VM configuration (Debian inside the VM)

auto lo
iface lo inet loopback

auto eth0
iface eth0 inet static
        address 192.168.0.2/24
        gateway 192.168.0.1
        dns-nameservers 1.1.1.1 8.8.8.8
Important tradeoff of this approach
With one public IP, every port can only map to one VM. You cannot have two VMs both listening on port 443. If you need multiple HTTPS services, you need a reverse proxy (like Nginx or Caddy) on one VM that routes by domain name — but that's a later lesson.

Interactive Quiz

Check your understanding

  1. Your Proxmox host has one public IP and three VMs behind NAT. VM1 downloads a file. The remote server sees the request coming from what IP?
    2. The Proxmox host's public IP (51.195.46.19). The MASQUERADE rule rewrote VM1's private source IP to the host's public IP.
  2. You want Nextcloud on VM1 (port 443) and a static site on VM2 (port 443). Can both VMs use port 443? Why or why not?
    3. No. With one public IP, the DNAT rule can only forward port 443 to a single destination. To serve multiple HTTPS apps, you'd use a reverse proxy on one VM that inspects the domain name and routes internally.
  3. You reboot your KimSufi server. The VMs come up but they can't reach the internet. What did you likely forget?
    4. You didn't persist the iptables rules. They're lost on reboot. Either use iptables-persistent or put the rules in /etc/rc.local.
  4. Your OVH gateway is 100.64.0.1, not 192.168.0.1. Why can't VMs use 100.64.0.1 as their default gateway?
    5. Because 100.64.0.1 is on the host's public network, not on the bridge. VMs are on the 192.168.0.0/24 bridge network. They must use 192.168.0.1 (the bridge IP) as their gateway. The host then handles masquerading and forwarding to the real gateway.
